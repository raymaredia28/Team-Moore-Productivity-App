TRACK 3 — TESTING & QA  (no coding required)
═══════════════════════════════════════════════════════
Good for: anyone — just needs a browser and the app running

Run through every item below and note: Pass / Fail / Looks off

Core flows:
  [ ] Add a task (full wizard) → appears in Today's Focus
  [ ] Complete a task → progress bar updates, flash message shows and auto-disappears
  [ ] Edit a task → wizard opens pre-filled, changes save correctly
  [ ] Delete a task → disappears immediately
  [ ] Enter Focus Mode → complete top task → next task auto-loads
  [ ] All Tasks: active / all / done filters each show correct tasks
  [ ] Progress view: complete tasks, check bar chart and category bars update
  [ ] Refresh page → tasks and settings still there (localStorage persistence)

Settings:
  [ ] Set name → greeting on dashboard updates
  [ ] Toggle dark mode → whole app switches instantly, no white flash
  [ ] Switch to Large text → all text scales up, nothing overflows
  [ ] Change daily limit to 5 → Focus view shows up to 5 tasks

Accessibility:
  [ ] Tab through entire app — every button should show a green focus ring
  [ ] Screen reader: check aria-labels make sense on task complete/edit/delete buttons
  [ ] Enable OS reduced motion → animations should collapse to near-instant
  [ ] Test at 150% browser zoom → nothing should overflow or break

Edge cases:
  [ ] Add task with only a title, skip all other steps
  [ ] Complete every single task → dashboard should say "All caught up!" not "No tasks yet"
  [ ] Delete every task → all views show friendly empty states, no crashes
  [ ] Very long task title (fill all 120 characters) → card should not overflow
  [ ] Open app on a phone screen (or DevTools 375px) → layout should still work

TRACK 4 — HCI WRITE-UP & DOCUMENTATION  (writing work)
═══════════════════════════════════════════════════════
Good for: someone working on the project report / presentation

The app was built specifically around the "Guided Minimalist Task Flow" concept.
README.md already has a full HCI mapping section — use it as a starting point.

Things to document / expand on:

  1. Map features to HCI heuristics
     Go through Nielsen's 10 usability heuristics and note where the app satisfies
     each one. Specific examples already in the app:
     - Visibility of system status → progress bar, completion flash message
     - User control & freedom → edit task, undo (once Track 2 adds it)
     - Consistency & standards → same wizard for add and edit, same card layout everywhere
     - Error prevention → optional fields, guided steps instead of freeform forms
     - Aesthetic & minimalist design → top-3 task limit, no sidebar, no widget grid

  2. Describe the Add Task wizard design rationale
     The wizard was intentionally built as one question per step (not a full form).
     This reduces cognitive load per Hick's Law — more choices at once = longer decision time.
     Worth citing in the write-up.

  3. Accessibility section
     App includes: keyboard navigation (focus rings), aria-labels, aria-pressed, aria-current,
     dark mode (low vision / eye strain), large text mode, reduced motion support.
     Document these for the accessibility requirements section of the report.

  4. Known limitations to acknowledge
     - No backend / multi-device sync (intentional for prototype, noted in README)
     - Accent color setting exists in UI but doesn't propagate yet (Track 2 item)
     - Mobile hover issue on task cards (Track 1 item)
     Acknowledging limitations honestly is standard for HCI prototypes.

  5. Suggested user study questions (if doing evaluation)
     - "How long did it take you to add your first task?"
     - "Did you feel overwhelmed at any point? Where?"
     - "Did the encouraging messages feel genuine or generic?"
     - "What would make you actually use this app daily?"
